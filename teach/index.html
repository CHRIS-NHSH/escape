用 AI 做出闖關機關

如何用 AI 做出像範例庫那樣的互動機關。重點不在會不會叫 AI 生程式，而在想不想得清楚一個好玩的機關到底該長什麼樣。

← 回機關範例庫
核心觀念 製作步驟 設計規格 提示詞產生器 好習慣 發佈分享

①核心觀念

與其叫 AI 從零生出一整個遊戲，不如先把設計想清楚，再請它實作。為什麼？因為 AI 生出來的東西「看起來會動」不代表真的對——要做的，是把想法講清楚、再一一驗收。這要訓練的，是兩件帶得走的能力：

一、把設計想清楚，再開口問 AI。先寫好規格，AI 才生得準（見下方規格）。

二、會驗收自己的成果。會測、會回報問題、能說明自己做了什麼。

一句話：別只想著「叫 AI 生一個給我」，而是先當設計師、再當品管。這兩件事換任何 AI 工具、任何題目都用得上。

②一個機關怎麼做出來

先看整個流程，後面幾段再一一細談。

  1. 玩 + 拆解：先玩現有機關，拆解它的玩法、規則與風格——也就是下面要教的「規格」。
  2. 寫規格：把想做的機關，規格填清楚——這一步先別碰 AI。
  3. 用 AI 實作：把規格 + 技術要點用下面的產生器組成提示詞，貼給 AI，邊生邊測。
  4. 互測 + 展示：跟同學交換玩、回報問題、修好、再展示。機關本來就是要被別人玩的。

③設計規格

範例庫裡的機關都是同一個結構。動手前先把這份規格想清楚，先別急著找 AI。

玩法是什麼？
玩家看到什麼、要做什麼動作？
怎樣算過關？
一個明確、不含糊的成功條件。
過關給什麼線索？
成功後顯示的文字或圖片。
風格與氛圍？
美術風格、背景、配色、字體感覺——例如老舊羊皮紙、科幻面板、可愛手繪。這決定整個機關的「長相」，最容易被忘記，卻最影響第一眼。
後台能調什麼？進階選用
後台是為了之後能快速修改、管理（換答案、改題目、調時間），對玩遊戲不是必要的。先做出能玩的版本，行有餘力再加。

④規格 → 提示詞產生器

把規格填一填，下面的技術要點建議全勾（它們會附在提示最後），按「產生提示詞」就會組出一段可直接貼給 AI 的提示。這就是把設計變成 AI 看得懂的需求。

這幾條 AI 容易漏、漏了又不容易發現，每條都標了「沒寫會怎樣」。

⑤要養成的習慣

  • 驗證，不盲信。AI 給的程式，每個功能都自己實測：手機點得動嗎？答錯有反應嗎？後台存得了嗎？奇怪的輸入會不會壞？
  • 會回報問題。講「具體現象」而不是只說「壞了」。試試這個句型：
    「我點了 ＿＿，預期會 ＿＿，結果卻 ＿＿。」這樣 AI 才修得準。
  • 需求講滿、工作切小。一開始把要的東西和限制一次講清楚，沒講的 AI 會自己猜；後續修改一次只改一小塊，改一段、驗一段，才抓得到錯、保持得住掌控。這不只對 AI，做任何事都適用。

⑥做好之後：發佈與分享

機關做好是一個 HTML 檔。要讓別人也能玩，得放到網路上。兩個免費又常用的方法：

GitHub Pages　純前端、最簡單

把 HTML 檔放進 GitHub 倉庫、開啟 Pages，就得到一個網址。改完重新上傳就更新。沒有後端、不會存資料。

建倉庫 → 上傳 index.html → Settings → Pages → 選 main /(root) → 得到 帳號.github.io/倉庫名

→ 註冊 GitHub

GAS（Google Apps Script）　需要後端時

用 Google 帳號把網頁部署成 Web App。當你要把分數、設定存到 Google 試算表，或想跨裝置共享進度時用它。

script.google.com 新增專案 → 寫一個 doGet() 回傳 HtmlService → 部署 → 網頁應用程式 → 取得網址

→ 開啟 Apps Script　需 Google 帳號

怎麼選：先求「能玩、純前端」就用 GitHub Pages；要存分數、共享設定、跨裝置同步，再用 GAS。如果是從範例庫的範本改的，後台還有「產生分享連結」——不用後端，也能把「這一份設定」用一條網址分享出去。
